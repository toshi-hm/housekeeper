# Claude Code Skills — 設計方針・実装方針

このリポジトリに同梱する Claude Code skills の設計記録。
skills の実体は `.claude/skills/<name>/` に置く。

## 1. 目的

- **開発の型化**: コンポーネント作成・テスト・リファクタリングなどの定型作業を、
  どのリポジトリでも同じ品質で再現できるようにする
- **レビューの型化**: UI/UX・PWA など「観点の抜け漏れ」が起きやすい領域をチェックリスト化する
- **企画の型化**: 新機能提案 → spec ドラフト → ロードマップ反映、の一連の流れを再現可能にする
- **生活への活用**: 献立・在庫・旅行・家計・掃除・文書作成など、開発以外の実生活タスクを支援する

## 2. ポータビリティ設計（2層構造）

**スキルは他のリポジトリへコピーして再利用できることを前提に設計する。**
そのため各スキルを「汎用コア」と「プロジェクト設定」に分離する。

```
.claude/skills/
  <skill-name>/
    SKILL.md        # 汎用コア。特定リポジトリの固有名詞を含まない
    PROJECT.md      # プロジェクト設定（任意）。このリポジトリ固有の値・規約・パス
docs/skills/
  README.md         # 本ドキュメント（設計方針 + カタログ）
```

### 2.1. SKILL.md（汎用コア）の規約

- **特定リポジトリの固有名詞（パス・コマンド・テーブル名・spec 名）を書かない**
- 冒頭に必ず「Step 0. プロジェクト設定の解決」を置く:
  1. 同ディレクトリに `PROJECT.md` があれば読み込み、その設定を最優先する
  2. なければリポジトリの `CLAUDE.md` / `package.json` / lint 設定などから自動検出する
  3. 検出もできない項目は SKILL.md 内のデフォルト表に従う
- 手順・チェックリスト・アンチパターンはどのプロジェクトでも成立する一般形で書く

### 2.2. PROJECT.md（プロジェクト設定）の規約

- SKILL.md が参照する設定項目（テストランナー、コマンド、規約、参照すべき spec、
  データソース等）を、このリポジトリの実値で埋める
- コード例はそのリポジトリの実在コードのパターンから引く（架空の API を書かない）
- PROJECT.md が SKILL.md の一般論と矛盾する場合は **PROJECT.md が勝つ**

### 2.3. 他リポジトリへの持ち出し方

1. `.claude/skills/<name>/` ディレクトリごとコピーする
2. `PROJECT.md` を削除する（→ 自動検出モードで動く）か、移植先の値で書き直す
3. 移植先の CLAUDE.md にスキル一覧を追記する（発動率が上がる）

生活系スキルの多くは PROJECT.md 不要（完全汎用）なので、コピーだけで動く。

## 3. frontmatter 規約

```yaml
---
name: <kebab-case のスキル名>
description: >-
  いつ発動すべきかを書く。日本語のトリガーフレーズ例を必ず含める
  （ユーザーは日本語で依頼するため）。
---
```

- `description` は「何をするか」ではなく **「いつ使うか」** を主体に書く
  （Claude Code はこの文でスキル選択を判断する）
- 日本語トリガー例（「〜して」「〜をレビュー」）を明記する
- description にも特定リポジトリの固有名詞を書かない（持ち出し時に書き換え不要にする）

## 4. 本文の書き方

- 手順は Step 形式で番号付けし、**判断が必要な箇所には判断基準（表・条件）を添える**
- 「やってはいけないこと」（アンチパターン）を明示する
- 完了条件（Definition of Done）を末尾に置く
- コミットの実施可否はスキル側で規定せず、ユーザー指示とセッションの文脈に従う
- データを扱う生活系スキルは **読み取り専用** を原則とし、書き込みは
  「内容を提示 → ユーザー承認 → 実行」に限定する

## 6. 実装方針（各スキルの設計判断）

### react-component

- **入口は分類判断**: 実装より先にプロジェクトのコンポーネント分類規約
  （デフォルト: Atomic Design）に当てて配置層を決めさせる
- Story とテストをコンポーネントと**同時に**作らせる（「後で書く」を許すと規約違反が常態化する）
- housekeeper の PROJECT.md には Atomic Design の層別ルール・実在コード準拠のテンプレート・
  i18n 規約を置く

### ts-quality

- 汎用コアは「strict / no any / unknown+バリデーション / 網羅性チェック」などの一般原則
- 記法の好み（interface vs type、アロー関数強制など）はプロジェクトごとに割れるため
  **すべて PROJECT.md 側**に置く

### unit-test

- **最重要の設計判断**: テストランナーを仮定しない。Step 0 で必ず特定させる
  （AI が vitest の API を bun test のリポジトリに書く事故が最も起きやすい）
- 汎用コアはランナー横断で通用するパターン（AAA、境界値、モック方針、colocation）
- housekeeper の PROJECT.md に bun test / happy-dom / i18n wrapper / Supabase mock の実パターンを置く

### pwa-doctor

- 診断チェックリスト（インストール可能性 / オフライン / キャッシュ / SW 更新 / Push）は汎用
- 「そのプロジェクトのオフライン方針（参照のみ？キューイングあり？）」は PROJECT.md で解決する

### uiux-review

- モバイルファースト・状態網羅・a11y・フォーム UX のチェックリストは汎用
- 「主戦場の利用文脈（housekeeper ならキッチンで片手操作）」を PROJECT.md から注入し、
  指摘を文脈化する

### feature-proposal

- 提案 → 承認 → spec ドラフト → ロードマップ追記、の段階制ワークフローは汎用
- プロダクト制約（単一ユーザー・バックエンドなし等）・spec テンプレート・
  ロードマップ文書の場所は PROJECT.md で解決する

### recipe-from-stock / pantry-review（在庫連携）

- 在庫データの取得手段を Step 0 で解決する: PROJECT.md にデータソース定義
  （housekeeper なら Supabase MCP + SQL）があればそれを使い、
  なければユーザーに在庫を貼ってもらう完全汎用モードで動く
- **読み取り専用**。書き込みは提示 + 承認制
- 期限接近アイテムの消費を最優先する（食品ロス削減）

### travel-planner / household-budget / cleaning-planner / formal-writing（完全汎用）

- リポジトリのコード・データに一切依存しない。PROJECT.md なし
- いずれも「まずヒアリング項目を絞って確認 → 構造化された成果物を出す」型で統一
- 個人情報（家計・宛先）を扱うため、外部送信・保存をしない旨を明記する

## 7. メンテナンス方針

- プロジェクトの規約・spec を変更したら、対応する **PROJECT.md** を更新する
  （SKILL.md は汎用なので原則触らない。汎用手順自体の改善時のみ更新）
- スキルにコード例を足すときは、実在コードから引く（動かない例を置かない）
- スキルの追加・削除時は本ドキュメントのカタログと CLAUDE.md の一覧を更新する
- 他リポジトリへ持ち出したスキルは持ち出し先で独立に育てる（このリポジトリとの同期は取らない）
