# Skill: uiux-review

UI/UX とアクセシビリティの観点でコンポーネント・画面をレビューする。
このアプリは **スマホ片手操作（キッチン・買い物中）が主戦場** — すべての判断をそこに合わせる。

## レビュー手順

1. 対象を確定する（指定がなければ直近の diff、それもなければユーザーに画面を確認）
2. 対象コンポーネントと、それを使う上位（organisms / pages）のコードを読む
3. 下のチェックリストを順に適用する
4. findings を出力形式に従って報告する（修正は指示があった場合のみ）

## チェックリスト

### 1. モバイルファースト（最優先）

- [ ] タッチターゲットが十分（目安 44×44px 以上。`h-11` 以上 / p 込みで確保）
- [ ] 主要アクション（消費・追加・スキャン）が親指の届く位置にあるか（画面下部優先）
- [ ] 基本スタイルがスマホ幅で、`sm:` `md:` は「広げる」方向にだけ使われているか
- [ ] 横スクロールが発生しない（固定幅・長い単語・テーブル）
- [ ] 片手・濡れた手・調理中でも押し間違えない間隔か（破壊的操作が他ボタンに隣接していないか）

### 2. 状態の網羅

- [ ] loading（Spinner / skeleton。レイアウトシフトしないか）
- [ ] error（再試行手段があるか。生のエラーメッセージを出していないか）
- [ ] empty(0件)（次の行動への導線があるか。「追加する」ボタン等）
- [ ] offline（参照は可・編集はトースト抑止、が守られているか）
- [ ] 楽観更新中 / 送信中の二重タップ防止（disabled + 表示）

### 3. フォーム UX

- [ ] `inputMode` / `type` が適切（数量: `inputMode="numeric"`、日付: date picker）
- [ ] バリデーションエラーが**フィールドの近く**に、送信時ではなく blur/入力時に出るか
- [ ] 必須/任意が見て分かるか
- [ ] スマホでキーボードが内容を隠さないか（送信ボタンの位置）
- [ ] 破壊的操作（削除）に確認ダイアログ（`ConfirmDialog`）があるか

### 4. アクセシビリティ

- [ ] アイコンだけのボタンに `aria-label`（i18n 済みの文字列で）
- [ ] 画像に `alt`
- [ ] 色だけで意味を伝えていない（期限バッジ: 色 + テキスト/アイコン）
- [ ] コントラスト比（本文 4.5:1、大文字 3:1）
- [ ] フォーカス可視（focus-visible スタイルが Tailwind で消されていないか）
- [ ] 見出し階層・ランドマークが妥当
- [ ] Storybook の `@storybook/addon-a11y` で警告が出ていないか

### 5. i18n

- [ ] ハードコード文字列がない（`t()` 経由。ja/en 両方にキーがある）
- [ ] 英語で文字列が伸びてもレイアウトが壊れない（truncate / wrap の指定）
- [ ] 日付・数値の表示がロケール対応か

### 6. 一貫性

- [ ] 既存の shadcn/ui・atoms を再利用しているか（似た見た目の別実装がないか）
- [ ] 余白・角丸・タイポグラフィが周辺画面と揃っているか
- [ ] トースト / ダイアログ / インライン、フィードバック手段の使い分けが周囲と一貫しているか

## 出力形式

```markdown
## UI/UX Review: <対象>

### Findings

| #   | Severity | 箇所                                     | 問題                                           | 提案                           |
| --- | -------- | ---------------------------------------- | ---------------------------------------------- | ------------------------------ |
| 1   | High     | src/components/molecules/ItemCard.tsx:42 | 削除ボタンが消費ボタンに隣接し誤タップしやすい | 間隔を空ける or メニューに格納 |

- **High**: 誤操作・データ喪失・操作不能につながる
- **Medium**: 使いにくいが回避可能
- **Low**: 磨き込み（polish）

### 良い点

（維持すべき点を 1〜3 個。次の変更で壊されないように明記する）
```

- findings には必ず `file:line` を付ける
- 「一般論としてダメ」ではなく「この画面のこの利用文脈でダメ」まで書く
- 修正まで依頼された場合は High → Medium の順に適用し、`react-component` /
  `ts-quality` スキルの規約に従って実装する

## やってはいけないこと

- デスクトップ前提の指摘（このアプリに hover 前提 UI を要求しない）
- 好みの言い換えを Medium 以上にする（根拠のない指摘は Low か、出さない）
- レビューのついでに無断でコードを書き換える（依頼が「レビュー」なら報告まで）
