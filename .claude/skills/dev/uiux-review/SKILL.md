# Skill: uiux-review

UI/UX とアクセシビリティの観点でコンポーネント・画面をレビューする。

## Step 0. プロジェクト設定の解決

1. **同ディレクトリの `PROJECT.md` があれば読み、そのプロダクトの利用文脈を最優先する**
2. なければ README / CLAUDE.md / spec から以下を把握する:
   - **主戦場のデバイスと利用シーン**（モバイル片手？ デスクトップ業務？ タブレット現場作業？）
   - デザインシステム / UI ライブラリ
   - i18n の有無と対象ロケール
   - オフライン・低速回線への配慮が必要か
3. 利用文脈が分からない場合はユーザーに一言確認する
   （文脈のないレビューは一般論の羅列になり価値が薄い）

## レビュー手順

1. 対象を確定する（指定がなければ直近の diff、それもなければユーザーに確認）
2. 対象コンポーネントと、それを使う上位（画面・ページ）のコードを読む
3. 下のチェックリストを利用文脈に照らして適用する
4. findings を出力形式に従って報告する（修正は指示があった場合のみ）

## チェックリスト

### 1. 利用文脈への適合（最優先）

- [ ] 主要アクションが主戦場のデバイスで最短距離にあるか
      （モバイルなら親指圏・画面下部、デスクトップならキーボード操作）
- [ ] タッチ対象が十分（目安 44×44px 以上）、誤タップしない間隔か
      （破壊的操作が他ボタンに隣接していないか）
- [ ] モバイルファーストのプロジェクトなら、基本スタイルが最小幅で
      ブレークポイントは「広げる」方向にだけ使われているか
- [ ] 横スクロールが発生しない（固定幅・長い単語・テーブル）

### 2. 状態の網羅

- [ ] loading（スピナー / skeleton。レイアウトシフトしないか）
- [ ] error（再試行手段があるか。生のエラーメッセージを出していないか）
- [ ] empty(0件)（次の行動への導線があるか）
- [ ] offline（プロジェクトのオフライン方針どおりか）
- [ ] 送信中の二重実行防止（disabled + 進行表示）

### 3. フォーム UX

- [ ] `inputMode` / `type` が適切（数値・電話・メール・日付）
- [ ] バリデーションエラーがフィールドの近くに、適切なタイミング（blur/入力時）で出るか
- [ ] 必須/任意が見て分かるか
- [ ] モバイルでキーボードが内容・送信ボタンを隠さないか
- [ ] 破壊的操作に確認ステップがあるか

### 4. アクセシビリティ

- [ ] アイコンだけのボタンに `aria-label`
- [ ] 画像に `alt`
- [ ] 色だけで意味を伝えていない（色 + テキスト/アイコン）
- [ ] コントラスト比（本文 4.5:1、大きい文字 3:1）
- [ ] フォーカス可視（focus スタイルが CSS で消されていないか）
- [ ] 見出し階層・ランドマークが妥当
- [ ] a11y 自動チェック（Storybook addon-a11y / axe 等、プロジェクトにあるもの）の警告

### 5. i18n（採用プロジェクトのみ）

- [ ] ハードコード文字列がない（全ロケールにキーがある）
- [ ] 文字列が長いロケールでもレイアウトが壊れない（truncate / wrap）
- [ ] 日付・数値の表示がロケール対応か

### 6. 一貫性

- [ ] 既存のデザインシステム・共通コンポーネントを再利用しているか（似た見た目の別実装がないか）
- [ ] 余白・角丸・タイポグラフィが周辺画面と揃っているか
- [ ] フィードバック手段（トースト / ダイアログ / インライン）の使い分けが周囲と一貫しているか

## 出力形式

```markdown
## UI/UX Review: <対象>

### Findings

| #   | Severity | 箇所          | 問題                     | 提案             |
| --- | -------- | ------------- | ------------------------ | ---------------- |
| 1   | High     | src/...tsx:42 | （利用文脈に即して書く） | （具体的な対策） |

- **High**: 誤操作・データ喪失・操作不能につながる
- **Medium**: 使いにくいが回避可能
- **Low**: 磨き込み（polish）

### 良い点

（維持すべき点を 1〜3 個。次の変更で壊されないように明記する）
```

- findings には必ず `file:line` を付ける
- 「一般論としてダメ」ではなく「この画面のこの利用文脈でダメ」まで書く
- 修正まで依頼された場合は High → Medium の順に適用し、プロジェクトの実装規約に従う

## やってはいけないこと

- 利用文脈と逆のデバイス前提の指摘（モバイル主戦場のアプリに hover 前提 UI を要求する等）
- 好みの言い換えを Medium 以上にする（根拠のない指摘は Low か、出さない）
- レビューのついでに無断でコードを書き換える（依頼が「レビュー」なら報告まで）
