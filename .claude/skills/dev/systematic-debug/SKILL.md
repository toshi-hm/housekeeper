# Skill: systematic-debug

バグ・不可解な挙動を体系的に調査して直す。
**当てずっぽうの修正を繰り返さない**ための型。

## Step 1. 症状を固定する

修正に入る前に、以下を明文化する（ユーザーの報告が曖昧なら、ここだけ確認する）:

- **期待**: 何が起きるべきか
- **実際**: 何が起きているか（エラーメッセージは原文ママ・全文で控える）
- **再現手順**: 最小の手順。「たまに」なら発生条件の候補（時刻・データ量・並行操作・環境）
- **範囲**: いつから（動いていた最後の時点）/ どの環境で（dev/prod、ブラウザ、OS）

**再現できない問題は直せたことを確認できない。** 再現が不安定なら、まず再現率を上げる
（ループ実行・データ固定・タイミング強制）ことに投資する。

## Step 2. 事実を集める（推測の前に）

- エラーの**全文とスタックトレース**を読む（先頭行だけで判断しない。cause チェーンも辿る）
- ログ・デバッガ・print で「どこまでは正しく動いているか」の境界を見つける
- `git log` / `git diff` — 動いていた時点との差分。回帰なら `git bisect` が最速のことが多い
- 環境差（Node/ランタイムのバージョン、環境変数、ビルドキャッシュ、依存の lockfile 差分)

## Step 3. 仮説 → 切り分けのループ

1. 事実から仮説を立てる（「◯◯が null になっている」レベルまで具体化）
2. **仮説を最小コストで検証する実験**を設計する（ログ 1 行、条件を固定した再現、二分探索）
3. 結果で仮説を棄却 or 絞り込み、次の仮説へ

### 切り分けの定石

- **二分探索**: コード（コメントアウト・early return）、データ（半分に削る）、
  時間（git bisect）、環境（別マシン・シークレットウィンドウ）
- **層の特定**: 入力 → 変換 → 出力のどの層で値が壊れるかを境界ごとに観測する
- **最小再現**: 疑わしい部分だけを抜き出した単体スクリプト/テストで再現させる

### ありがちな真犯人チェックリスト

- 非同期の競合（await 漏れ、順序依存、stale closure）
- キャッシュ（ビルドキャッシュ、HTTP キャッシュ、メモ化、Service Worker）
- 環境変数・設定の環境差 / undefined の黙殺（`??` や optional chaining の陰）
- タイムゾーン・ロケール・文字コード
- 依存のバージョン差（lockfile と node_modules の不整合 → クリーン再インストール）
- 「直したはずのコードが実行されていない」（別ファイル・別ビルド・デプロイ前）

## Step 4. 修正する

- **根本原因を直す**。症状の隠蔽（try/catch で握る、null チェックを盲目的に足す）で
  済ませない。対症療法しかできない事情があるなら、その旨と根本対策の TODO を明記する
- 修正は最小差分にする。「ついで直し」は分けて提案する
- 同型のバグが他にもないか、原因のパターンで横展開検索する（`rg` で同じ誤用を探す）

## Step 5. 回帰を防ぐ

- **バグを再現するテストを先に書き、修正で緑になることを確認する**（テスト基盤がある場合）
- 元の再現手順で症状が消えたことを確認する（テストが通っただけで完了にしない）
- 報告する: 原因（なぜ起きたか）/ 修正（何を変えたか）/ 検証（どう確かめたか）の 3 点セット

## やってはいけないこと

- 再現確認なしの「直ったはず」報告
- エラーメッセージを読まずに推測で書き換える / 当てずっぽう修正の繰り返し
- 原因不明のまま「リトライを足す」「タイムアウトを伸ばす」で症状だけ消す
- 調査で得た事実（ログ・差分）を報告に残さない（次に同じ調査を繰り返すことになる）
