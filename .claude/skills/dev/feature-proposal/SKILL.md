# Skill: feature-proposal

新機能の提案 → spec ドラフト → ロードマップ反映を段階的に行う。
**提案が承認されるまで、ファイルへの書き込みは一切しない。**

## Step 0. プロジェクト設定の解決

1. **同ディレクトリの `PROJECT.md` があれば読み、プロダクト制約・文書構成を最優先する**
2. なければ以下から把握する:
   - README / CLAUDE.md — プロダクトの目的・対象ユーザー・技術制約
   - ロードマップ / TODO 文書（PLANS.md / ROADMAP.md / Issues 等）— 既存計画
   - spec / 設計文書のディレクトリと、その見出し構成（新規 spec はこれを踏襲する）
3. 把握すべき最低ライン: **誰のための何のプロダクトか / 越えてはいけない技術制約 /
   提案が既存計画と被っていないか**

## Step 1. 現状把握

1. ロードマップ文書を読む — 既存 TODO・Backlog と被る提案をしないため
2. 関連する既存 spec / 実装を読む
3. データモデルを確認する — 既存スキーマで賄えるか、変更が要るか

## Step 2. 提案を作る

1 機能につき以下のフォーマットで提示する。複数案を求められたら 3〜5 案を
短縮版（課題・解決策・工数）で出し、深掘りする案を選んでもらう。

```markdown
## 提案: <機能名>

### 課題（いま何が不便か）

（そのプロダクトの実際の利用シーンに即して書く）

### 解決策

（ユーザーから見える振る舞い。UI のラフ説明を含む）

### スコープ

- やること: ...
- やらないこと: ...（意図的に外すものを明記）

### データへの影響

- 既存スキーマで可 / 変更が必要（概要と、削除・整合性の方針）

### 技術ポイント

- プロダクトの技術制約との整合、外部連携の要否、既存機能との相互作用

### 工数目安

S（〜半日）/ M（1〜2日）/ L（3日〜）

### 優先度・時期の提案

（プロジェクトのフェーズ/マイルストーン体系に沿って + 理由）
```

### 提案の質の基準

- プロダクトの技術制約・対象ユーザーに反する提案はしない。制約内に収まる代案に変形するか、
  「制約を変えるコストに見合うか」を明示して問う
- 「やらないこと」を必ず書く（スコープの膨張を防ぐ）
- 工数・優先度のない提案を出さない（判断材料にならない）

## Step 3. 承認後 — spec ドラフトを書く

- プロジェクトの spec ディレクトリに、**既存 spec の見出し構成を踏襲して**新規作成する
  （概要 / スコープ判断 / ユーザーストーリー / 実装方針 / データ / エラー / 対象範囲、が一般形）
- 実装方針にはコンポーネント・モジュールの配置方針まで書く
- spec の索引（CLAUDE.md の一覧等）があれば 1 行追加する

## Step 4. ロードマップへ反映する

- ロードマップ文書の該当フェーズ配下に、**1 PR で完結する粒度**でタスクを分解して追記する
- タスクは依存順（データ → ロジック → UI → テスト）に並べる
- Issue 起票・チケット化が別スキル/別フローの責務なら、そちらに委ねる（勝手に起票しない）
- 重要な判断（スコープ外にしたもの等）は決定ログがあればそこにも追記する

## Step 5. 報告

提案の要約、作成/変更したファイル一覧、次のアクション（起票、実装開始の指示待ち）を報告する。

## やってはいけないこと

- 承認前の spec 作成・ロードマップ変更
- 既存計画と重複する提案（Step 1 を飛ばさない)
- プロダクトの技術制約を無視した提案（「バックエンドを立てれば簡単」系）
- 工数・優先度を書かない提案
