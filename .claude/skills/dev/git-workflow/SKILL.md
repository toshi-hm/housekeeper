# Skill: git-workflow

変更をレビューしやすい形に整える: コミット分割・メッセージ・PR 説明。

## Step 0. プロジェクト規約の解決

1. コミット規約の検出: `commitlint.config.*` / `.commitlintrc*` があれば **Conventional Commits 強制**。
   なければ `git log --oneline -20` で既存メッセージのスタイル（言語・prefix 有無）に合わせる
2. ブランチ命名: 既存ブランチ（`git branch -r`）の支配的パターンに合わせる
3. PR テンプレート: `.github/pull_request_template.md` 等があれば構成を踏襲する
4. CLAUDE.md 等に「コミット前に必ず実行するチェック」があれば、コミット前に必ず回す

## コミットの原則

### 分割の単位 = 「1 コミット 1 目的」

- **混ぜない**: 機能追加 + リファクタ + フォーマット修正は別コミットに分ける
  （フォーマット一括適用は単独コミットにしないと diff レビューが死ぬ）
- **各コミットでビルド・テストが通る**状態を保つ（bisect 可能な履歴）
- 分割の目安: 「このコミットだけ revert できるか？」

### 作業ツリーが混ざってしまっている場合

1. `git status` + `git diff` で変更を棚卸しし、論理単位に分類する
2. `git add -p`（hunk 単位のステージ）で 1 目的ずつコミットする
3. ファイル内で目的が混ざって分離不能なら、その旨を報告して 1 コミットにする
   （無理な履歴改変で内容を壊すより正直な方がよい）

### メッセージ（Conventional Commits の場合）

```
<type>(<scope>): <要約 — 変更の「何を」を命令形で>

<本文 — 「なぜ」を書く。コードを見れば分かる「どうやって」は書かない>
```

- type: `feat` / `fix` / `refactor` / `chore` / `docs` / `test` / `perf` / `ci` / `build`
- 要約は 50〜72 字目安。「Fix bug」「update」のような無情報メッセージを書かない
- 破壊的変更は `!` + 本文に `BREAKING CHANGE:` を明記

## 履歴の整理（rebase / squash）

- **push 済み共有ブランチの履歴は書き換えない**（自分だけのブランチに限る）
- WIP・typo 修正コミットの連なりは、レビュー前に論理単位へ squash / fixup する
- rebase 前に現在の HEAD を控える（`git log -1 --format=%H`）— 失敗時は reflog で戻れる
- コンフリクト解消時は「両方の意図」を理解してから解消する（機械的に片側を取らない）

## PR の作成・説明文

### 説明文の構成（テンプレートがなければ）

```markdown
## 概要

（この PR が解決する問題と、選んだアプローチ。1〜3 文）

## 変更内容

（レビュアーが diff を読む前の地図。箇条書き）

## 検証

（何をどう確認したか: テスト・手動確認・スクリーンショット）

## レビューの観点 / 影響範囲

（重点的に見てほしい箇所、リスク、やらなかったこと）
```

### PR の質の基準

- **サイズ**: 目安 400 行以下。超えるなら分割を提案する（依存順に stacked でも可）
- タイトルはマージ後の履歴で意味が分かる形（Conventional Commits 採用ならその形式で）
- 「なぜ」を説明文に書く（コードに書けない文脈こそ PR 説明の価値）
- スクリーンショット / 録画（UI 変更時）、Breaking / migration 手順（あれば）

## やってはいけないこと

- 指示なしの force push・共有ブランチの履歴改変
- `git add .` での無差別コミット（意図しないファイル混入。必ず `git status` で確認）
- 規約チェック（commitlint・pre-commit hook）の `--no-verify` スキップ
- 秘密情報（.env・鍵・トークン）のコミット。見つけたら**即報告**
  （コミット済みなら履歴からの除去 + 鍵ローテーションまで必要と伝える）
- 「とりあえず commit」の無情報メッセージ
