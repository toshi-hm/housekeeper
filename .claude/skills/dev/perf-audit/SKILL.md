# Skill: perf-audit

パフォーマンスの診断と改善を行う。**計測なしに最適化しない**が鉄則。

## Step 0. プロジェクト把握

- ビルドツール（Vite / webpack 等）とバンドル分析手段の有無
- 計測環境: 本番ビルドで測れるか（dev ビルドの計測値は信用しない）
- ユーザーの体感がどこで悪いのか: 初期表示 / 操作への反応 / スクロール / データ更新

## Step 1. 計測してボトルネックを特定する

症状に応じた計測手段を選ぶ:

| 症状           | 計測手段                                                            |
| -------------- | ------------------------------------------------------------------- |
| 初期表示が遅い | Lighthouse（本番ビルド）、Network タブ（waterfall）、バンドル分析   |
| 操作が重い     | Performance タブで録画 → Long Task / 何にかかっているかを見る       |
| 再レンダリング | React DevTools Profiler（何が・なぜ・何回描画されたか）             |
| メモリ・リーク | Memory タブのヒープスナップショット比較                             |
| API が遅い     | Network タブの TTFB（フロントの問題かサーバの問題かをまず切り分け） |

計測結果から **上位 1〜3 個のボトルネック**を数値付きで特定してから改善に入る。

## Step 2. 改善する（費用対効果順）

### 初期ロード

1. **不要なものを送らない**: バンドル分析で大物を特定 → 未使用依存の削除、
   重いライブラリの軽量代替、tree-shaking を壊す import の修正
2. **遅延する**: ルート単位の code splitting、モーダル・チャート等の重い UI の dynamic import、
   画像の遅延読み込み + 適切なフォーマット/サイズ
3. **先に出す**: クリティカル CSS、プリロード/プリフェッチ、スケルトン表示

### 実行時

1. **無駄な再レンダリングを減らす**: 状態を使う場所の近くに下ろす（state colocation）、
   リストの安定 key、context の分割。memo 化は「計測で効果を確認できる場合」に限る
2. **重い計算を移す**: メモ化、Web Worker、仮想スクロール（大量リスト）
3. **I/O を減らす**: リクエストの重複排除・キャッシュ（TanStack Query 等)、
   debounce/throttle、ページネーション

### データ層

- N+1 リクエスト、over-fetching（不要カラム・全件取得）、直列 await の並列化
  （`Promise.all`）を先に疑う — フロントの micro-optimization より効くことが多い

## Step 3. 効果を検証する

- **同じ条件で再計測**し、Before / After を数値で示す（例: LCP 3.8s → 2.1s、bundle 480KB → 310KB）
- 改善と引き換えに悪化していないか確認する（コード複雑化、キャッシュによる stale 表示、
  分割による遅延のちらつき）
- 効果が小さかった変更は**戻す**（複雑さだけ残さない）

## 報告形式

```markdown
## Perf Audit: <対象>

### 計測結果（ボトルネック）

1. <数値付きの事実>（例: vendor chunk 480KB、うち chart ライブラリ 210KB）

### 実施した改善と効果

| 改善 | Before | After |
| ---- | ------ | ----- |

### 未実施の提案（費用対効果順）
```

## やってはいけないこと

- 計測せずに memo / useCallback を撒く（読みにくくなるだけの場合が多い）
- dev ビルド・キャッシュ有効状態での計測値を根拠にする
- 数値の Before/After なしに「速くなった」と報告する
- 体感に影響しない箇所の最適化に時間を使う（ボトルネック上位から着手する）
